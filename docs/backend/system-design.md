# 系统设计

系统设计考察的是设计大型分布式系统的能力。

## 基础概念

### 可扩展性

| 方式 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **垂直扩展** | 升级单机配置（CPU、内存） | 简单，不改代码 | 有上限，成本非线性增长 |
| **水平扩展** | 增加机器数量 | 理论无上限，成本线性 | 需处理分布式问题 |

**原则：优先设计水平扩展方案。**

### 可用性计算

| SLA | 年停机时间 | 典型系统 |
|:---:|:---------:|---------|
| 99% | 87.6 小时 | 内部工具 |
| 99.9% | 8.76 小时 | 一般商业产品 |
| 99.99% | 52.6 分钟 | 金融、电商 |
| 99.999% | 5.26 分钟 | 关键基础设施 |

### 一致性模型

| 模型 | 描述 | 例子 |
|------|------|------|
| **强一致性** | 写入后立即可读 | 银行转账 |
| **最终一致性** | 一段时间后一致 | DNS、CDN |
| **因果一致性** | 有因果关系的操作有序 | 评论系统 |

::: tip CAP 定理
分布式系统只能在 **一致性 (C)**、**可用性 (A)**、**分区容错性 (P)** 中三选二。实际上 P 必须保证，所以是在 CP 和 AP 之间选择。
:::

## 核心组件

### 缓存

```text
浏览器缓存 → CDN 缓存 → 反向代理缓存 → 应用缓存 → 数据库
                    (Redis/Memcached)
```

**缓存策略：**
| 策略 | 工作机制 | 适用场景 |
|------|---------|---------|
| **Cache-Aside** | 应用同时维护缓存和 DB | 最常见的模式 |
| **Read-Through** | 缓存层负责从 DB 加载 | 数据访问较固定 |
| **Write-Through** | 写 DB 时同步写缓存 | 一致性要求高 |
| **Write-Behind** | 异步批量写 DB | 写操作频繁 |

**缓存三大问题及解决方案：**
| 问题 | 现象 | 解决方案 |
|------|------|---------|
| **穿透** | 查不存在的数据，绕过缓存直击 DB | 布隆过滤器 / 缓存空值 |
| **击穿** | 热点 key 过期，高并发打到 DB | 互斥锁 / 提前续期 |
| **雪崩** | 大量 key 同时过期，DB 被打垮 | 随机过期时间 / 多级缓存 |

### 消息队列

| 特性 | Kafka | RabbitMQ | Redis Stream |
|------|-------|----------|-------------|
| 吞吐量 | **极高**（百万/秒） | 万/秒 | 十万/秒 |
| 消息可靠性 | 高（持久化 + 副本） | 高 | 中 |
| 消息顺序 | 分区内有序 | 单队列有序 | 单消费者组有序 |
| 典型场景 | 日志收集、事件流、大数据 | 任务分发、可靠通知 | 轻量消息、实时流 |

### 负载均衡

**常见算法：**
| 算法 | 描述 | 适用场景 |
|------|------|---------|
| 轮询 (Round Robin) | 轮流分配 | 各节点能力相同 |
| 最少连接 | 分配给连接数最少的节点 | 长连接服务 |
| IP Hash | 按客户端 IP 哈希分配 | 需要会话粘滞 |
| 加权 | 按权重分配（性能强的权重高） | 异构服务器集群 |

## 经典系统设计案例

### 短链接系统

```text
需求：将长 URL 转为短链接，访问时重定向

核心设计：
1. 发号器生成唯一 ID（雪花算法 / 数据库自增）
2. Base62 编码 ID → 短码（6-7 位字符）
3. 存储到 Redis + DB（短码 → 长 URL）
4. 访问时 301/302 重定向

估算：
- 日活 1 亿，每用户 1 条链接 → 1 亿/天 ≈ 1150 QPS
- 存储 10 年 → 3650 亿条 → 约 10TB
```

### 秒杀系统

```text
挑战：瞬间高并发（库存有限，大量请求）

设计策略：
1. 前端限流 — 按钮置灰、验证码
2. 请求预处理 — Nginx 限流、API 网关过滤
3. 消息队列削峰 — 异步处理订单
4. 库存预热 — Redis 预减库存
5. 数据库减扣 — 乐观锁更新

流程：
用户 → Nginx 限流 → API 网关 → Redis 预减 → MQ 排队 → DB 落单
```

### 实时聊天系统

```text
架构：
- WebSocket 维持长连接（实时推送）
- 消息持久化到 DB（离线消息）
- 消息队列解耦（接收 → 处理 → 推送）

关键设计：
1. 连接管理：每个用户连接信息存 Redis
2. 消息 ID：全局递增（维持顺序）
3. 未读数：Redis 计数器
4. 多端同步：消息同步戳（sync cursor）
```

## 系统设计面试框架

```text
1. 明确需求（功能 + 非功能）
2. 估算规模（QPS、存储量、带宽）
3. 数据模型（Schema 设计）
4. 高层架构（画出系统框图）
5. 核心组件选型（DB、Cache、MQ 等）
6. 深入细节（每个模块的详细设计）
7. 扩展与优化（瓶颈分析、改进方向）
```
